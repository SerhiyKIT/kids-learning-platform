# Технічний фундамент

> **Статус:** зафіксоване рішення. Завершує критичний блок документів: «Бачення проекту» → «Формат подання уроків та ШІ» → «Модель даних» → «Обсяг MVP» → «Ролі та приватність» → цей документ.
>
> **Зафіксовані вибори:** бекенд — **Java + Spring Boot (Spring Security)**; фронтенд — **React (Next.js) + Capacitor (Android) + Tauri (desktop)**; хостинг — **VPS у ЄС (Hetzner/OVH) + Docker**.

---

## 1. Загальна схема

```
              ┌─────────────────────────────────────────────┐
              │              КЛІЄНТИ (один код React)        │
              │  Web/PWA (браузер) · Android (Capacitor) ·   │
              │  Desktop Windows/Linux/macOS (Tauri)         │
              └──────────────────────┬──────────────────────┘
                                     │ HTTPS / REST API (JSON)
              ┌──────────────────────▼──────────────────────┐
              │        БЕКЕНД: Java 21 + Spring Boot         │
              │  Spring Security (сесії, OAuth2 Google,      │
              │  ролі, права на кожному запиті)              │
              │  Модулі: auth · users · groups · lessons ·   │
              │  attempts · consents · admin · ai-pipeline   │
              └───────┬──────────────┬──────────────┬────────┘
                      │              │              │ (лише сервер!)
              ┌───────▼──────┐ ┌─────▼──────┐ ┌─────▼─────────────┐
              │ PostgreSQL   │ │ Об'єктне   │ │ Зовнішні API:      │
              │ (модель      │ │ сховище    │ │ LLM (уроки) ·      │
              │  даних)      │ │ (аудіо,    │ │ TTS (озвучення)    │
              │              │ │  ассети)   │ │                    │
              └──────────────┘ └────────────┘ └────────────────────┘
                    Усе на VPS у ЄС, у Docker-контейнерах
```

Принцип: **web-first**. Основний продукт — веб-застосунок (PWA); Android- і desktop-збірки — обгортки того самого коду, додаються після стабілізації веб-версії.

---

## 2. Бекенд

| Складова | Рішення | Примітки |
|---|---|---|
| Мова / платформа | **Java 21 (LTS) + Spring Boot 4.1** | зрілість, довгострокова підтримка; Spring Boot 3.x на момент переходу вже end-of-life |
| Безпека | **Spring Security** | сесії/JWT, OAuth2-клієнт для входу через Google, ролі (parent/teacher/admin), правила доступу на рівні методів (`@PreAuthorize`) |
| Доступ до БД | Spring Data JPA + **Flyway** (міграції) | схема БД версіонується в коді, за «Моделлю даних» |
| API | REST (JSON), документація **OpenAPI/Swagger** | контракт для всіх трьох клієнтів |
| Пошта | Spring Mail + SMTP-сервіс | підтвердження email, відновлення паролю, запрошення |
| Фонові задачі | Spring Scheduler | автоочищення історії (2 роки), стискання даних, розсилка нагадувань |
| ШІ-конвеєр | окремий модуль бекенда | виклики LLM (генерація JSON-сценаріїв) і TTS (пакетне озвучення); **ключі API — лише тут**, у клієнтський код не потрапляють ніколи |

Правила безпеки (з «Ролей та приватності», обов'язкові):
- перевірка прав на **кожному** ендпоінті (а не лише приховування кнопок в UI);
- паролі — bcrypt/argon2; 2FA (TOTP) для адмінів;
- audit log доступу адмінів до персональних даних;
- rate limiting на автентифікацію та публічні ендпоінти.

## 3. Фронтенд

| Складова | Рішення | Примітки |
|---|---|---|
| Фреймворк | **React + Next.js (TypeScript)** | один код для всіх платформ |
| PWA | Service Worker + маніфест | встановлення з браузера, **офлайн-кеш призначених уроків та аудіо** (вимога MVP) |
| Android | **Capacitor** | пакує веб-код у APK/AAB для Google Play; доступ до нативних API (у майбутньому — мікрофон) |
| Desktop | **Tauri** | легкі збірки для Windows/Linux/macOS |
| Рушій сцен | **SVG + CSS-анімації + Lottie** | за «Форматом подання уроків»; рендерить JSON-сценарії; заміна на PixiJS можлива пізніше без зміни сценаріїв |
| Стан | React Query (серверні дані) + легкий локальний стан | без зайвих бібліотек |
| UI | Tailwind CSS | швидка розробка; дитячий режим — власні великі компоненти за UX-гайдом |

Порядок платформ: **1) web/PWA (весь MVP) → 2) Android-збірка → 3) desktop-збірка.** Обгортки не починаються, доки веб-версія не пройшла тест із дітьми.

## 4. Дані та сховище

| Складова | Рішення |
|---|---|
| База даних | **PostgreSQL 16** (Docker-контейнер на VPS), схема — за «Моделлю даних», RLS як додатковий шар для таблиць із даними дітей |
| Файли (аудіо, ассети, Lottie) | S3-сумісне об'єктне сховище: **MinIO** на тому ж VPS (MVP) або керований S3 у ЄС (Hetzner Object Storage) — за єдиним S3-API, міграція прозора |
| Кеш/черги | не в MVP; за потреби — Redis |

---

## 5. Хостинг та експлуатація

| Складова | Рішення |
|---|---|
| Сервер | **VPS у ЄС** (Hetzner або OVH), від 2 vCPU / 4 GB для MVP |
| Контейнеризація | **Docker + docker-compose**: nginx (reverse proxy, TLS) · бекенд · PostgreSQL · MinIO |
| TLS | Let's Encrypt (автопродовження) |
| Фронтенд-хостинг | статична збірка Next.js за тим самим nginx (MVP); за потреби — CDN пізніше |
| Бекапи | щоденний дамп PostgreSQL + копія сховища на **окреме** сховище; цикл 35 днів (за «Ролями та приватністю»); щомісячна перевірка відновлення |
| Моніторинг | uptime-пінг + логи (достатньо для MVP) |
| Середовища | dev (локально, docker-compose) і prod (VPS); staging — після MVP |
| CI/CD | GitHub Actions: збірка, тести, деплой на VPS |

Юрисдикція: сервер і бекапи — **лише в дата-центрах ЄС** (вимога приватності).

## 6. Зовнішні сервіси

| Сервіс | Кандидати | Статус |
|---|---|---|
| LLM (генерація уроків) | Claude / GPT / Gemini | вибір за тестом якості української — **перший тиждень розробки** |
| TTS (озвучення) | ElevenLabs / Azure / Google / Polly | вибір за сліпим тестом голосів — **перший тиждень розробки** |
| SMTP (пошта) | будь-який транзакційний сервіс (напр. Resend, Postmark, SES) | обрати при реалізації auth |
| STT (голос) | не в MVP | паралельний прототип за «Обсягом MVP» |

Для всіх зовнішніх сервісів: виклики лише з бекенда; у клієнтах жодних ключів.

## 7. Репозиторії та якість коду

- **Монорепозиторій**: `/backend` (Java), `/frontend` (Next.js), `/infra` (docker-compose, конфіги), `/docs` (ці документи).
- Гілки: main (prod) + гілки задач; обов'язковий CI (збірка + тести) перед злиттям.
- Тести: юніт-тести критичної логіки (права доступу, згоди, каскадне видалення, валідація сценаріїв) — обов'язково; повне покриття — не мета MVP.
- Валідація JSON-сценаріїв: **єдина JSON-схема** зберігається в репозиторії, використовується і бекендом (перевірка згенерованого), і фронтендом (типи TypeScript генеруються з неї).

---

## 8. Свідомі компроміси (чесний список)

1. **Java-бекенд замість готової платформи** = максимальний контроль і зрілий security-стек, але фундамент (auth, сховище, деплой) будуємо самі — це найбільша частина роботи MVP.
2. **Все на одному VPS** (база, бекенд, файли) — просто й дешево для MVP; при зростанні перші кроки масштабування: винести файли на керований S3, потім базу на керований PostgreSQL.
3. **REST без GraphQL/gRPC** — простота важливіша за гнучкість на цьому етапі.
4. **Обгортки Capacitor/Tauri після MVP** — Google Play-версія не блокує перевірку гіпотез: PWA вже встановлюється на Android і desktop з браузера.

---

## 9. Рішення, зафіксовані цим документом

1. Бекенд: **Java 21 + Spring Boot 4.1 + Spring Security**; REST API; Flyway-міграції; ключі зовнішніх API — лише на сервері.
2. Фронтенд: **Next.js (React, TypeScript)**, web-first PWA; **Capacitor** для Android, **Tauri** для desktop — після стабілізації веб-версії.
3. Рушій сцен: **SVG + CSS + Lottie** (підтверджено).
4. Інфраструктура: **VPS у ЄС + Docker** (nginx, бекенд, PostgreSQL 16, MinIO), бекапи на окреме сховище, GitHub Actions.
5. Вибір LLM і голосу TTS — за зафіксованими процедурами тестування в перший тиждень розробки.
